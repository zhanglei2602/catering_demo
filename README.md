# 云汀私宴 Demo

这是一个围绕“私房菜饭店”场景制作的前后端一体 demo，当前包含两个页面：

- 菜单定制页：根据人数、预算、场景、偏好和忌口生成一套菜单
- 后厨调度页：模拟厨房持续运转、多桌并发和人工干预下的出餐调度

项目技术栈：

- Python：提供页面服务和菜单生成接口
- 原生 HTML + CSS + JavaScript：负责页面展示、动效和前端交互

## 启动方式

在项目根目录执行：

```bash
python app.py
```

启动成功后访问：

```text
http://127.0.0.1:8000
```

## 页面说明

### 1. 菜单定制页

地址：

```text
http://127.0.0.1:8000/
```

页面用途：

- 输入人数、用餐场景、预算风格、口味偏好、忌口和补充备注
- 生成一套更贴近私房菜节奏的菜单
- 展示每道菜的介绍、推荐理由、整桌结构和菜单表达

适合演示：

- 前台菜单推荐
- 私宴预订前的菜单沟通
- “根据条件生成菜单”的智能推荐场景

### 2. 后厨实时调度页

地址：

```text
http://127.0.0.1:8000/kitchen
```

页面用途：

- 配置厨房资源，例如厨师人数、冷菜台、热菜灶、炖煮炉、蒸箱
- 厨房持续运转，时间会一直推进
- 手动点击“加入随机一桌”，在当前时刻生成一桌新菜单并进入后厨流程
- 模拟多桌并行时的出餐调度，尽量避免只让一桌完整上满、其他桌长时间等待
- 支持人工干预，模拟真实门店里的突发情况

页面主要区域：

- `03 / 后厨工位`：看当前各个工位正在处理什么菜，也可以直接停用或恢复工位
- `04 / 前场桌面`：只显示当前仍在出餐中的桌台，并可直接催菜、加菜、退上一道重做、确认离场
- `05 / 桌台详情`：查看每桌进度、菜品状态和等待情况
- `06 / 出餐记录`：记录已经完成并人工确认离场的桌台，以及本桌实际出过的菜

当前已支持的人工干预：

- 厨师人数 `+1 / -1`
- 工位停用 / 恢复
- 催菜优先
- 加一道菜
- 退上一道重做
- 确认离场

适合演示：

- 后厨调度
- 多桌并发出餐
- 人工干预下的动态排程
- 餐厅运营看板和现场感展示

## 当前实现能力

- 内置一套模拟私房菜菜库，覆盖前菜、汤品、海鲜、禽类、肉菜、时蔬、主食、甜品
- 菜单生成会综合人数、预算、偏好、忌口和场景来选菜
- 后厨调度会区分冷菜、热菜、炖煮、蒸制等不同工位和耗时
- 调度逻辑会尽量兼顾各桌等待体验，而不是让某一桌持续独占出餐节奏
- 页面支持动态展示，可以看到时间推进、工位变化、前场桌台变化和出餐记录
- 页面支持人工干预，用于模拟营业过程中真实会发生的资源波动和订单变化

## 当前数据边界

当前项目中的菜单、菜品结构、耗时、桌台订单和后厨资源事件，都是为了演示交互和调度逻辑而构造的模拟数据。

这意味着当前 demo 适合：

- 演示产品思路
- 演示页面交互
- 演示调度算法方向
- 演示 AI 菜单推荐和后厨联动概念

这也意味着当前 demo 还不直接等价于真实门店系统，原因包括：

- 菜单数据不是来自真实门店的标准菜谱库
- 菜品介绍不是基于门店自己的菜品图文资产生成
- 耗时、工位占用、厨师需求目前是模拟估算
- 订单来源不是来自真实 POS、预订系统或门店运营系统
- 人工干预事件目前是前端手动模拟，不是来自真实业务流转

## 落地方案说明

如果要把这个 demo 往真实可用的方向推进，建议分成下面几步。

### 第一步：建设真实菜单与菜品基础数据

核心目标：把“模拟菜单”替换为“真实门店菜单资产”。

需要准备的数据：

- 真实菜单照片
- 每道菜的成品图
- 菜名、分类、售价、推荐人数
- 口味标签、忌口标签、过敏原信息
- 工艺标签，例如冷菜、热炒、炖煮、蒸制、甜品
- 预计制作时长
- 是否需要提前备料
- 是否占用特殊工位

推荐做法：

- 先整理门店现有菜单、菜牌、平时宣传图和后厨出品表
- 用真实菜单照片和菜品图建立标准化菜品库
- 对每道菜补充结构化字段，形成可计算的数据底座

### 第二步：接入 AI 菜单生成工作流

核心目标：让前台“根据人数和偏好生成菜单”的能力，真正建立在真实菜品库之上。

建议工作流：

1. 接收输入条件：人数、预算、场景、偏好、忌口、备注
2. 从真实菜品库中做规则过滤：先排除售罄、忌口、季节不符、工位冲突严重的菜
3. 将过滤后的候选菜品、门店风格信息、菜单照片和菜品图文信息送入 AI
4. 由 AI 组合出一套菜单结构，并生成对客展示文案
5. 输出菜单结果给前台页面，同时保留结构化字段给后厨调度使用

这里的关键不是只“接一个大模型接口”，而是要让 AI 建立在真实门店数据和业务规则之上。

### 第三步：接入真实订单与后厨排程工作流

核心目标：让后厨调度页从“本地模拟”变成“真实业务排程中台”。

建议接入的数据来源：

- 预订系统
- 点餐系统 / POS
- 服务员手持端
- 后厨叫号或出餐系统
- 原材料库存系统

建议工作流：

1. 前台生成菜单或顾客正式下单
2. 订单进入后厨任务队列
3. 系统根据菜品时长、工位类型、厨师资源、当前拥堵情况进行排程
4. 人工可以在现场进行催菜、加菜、退菜重做、工位停用、资源调整
5. 系统实时重算未开做任务的优先级
6. 出餐结果和人工干预记录反向沉淀，用于后续优化模型和排程参数

### 第四步：把“照片数据 + 结构化数据 + 工作流”真正 AI 化

建议 AI 化方向：

- 用真实菜单照片和菜品图，生成对客菜单文案和推荐话术
- 基于历史订单数据，学习不同人数和场景下的高满意度菜单组合
- 基于真实出餐耗时数据，动态修正每道菜的预计时长
- 基于催菜、退菜、加菜等历史事件，优化调度优先级模型
- 基于库存和售罄状态，动态调整推荐菜单和后厨排程

可以理解为：

- 真实菜单照片数据，解决“AI 说得像不像这家店”
- 真实结构化菜品数据，解决“AI 能不能算、能不能排”
- 真实工作流接入，解决“AI 能不能真正参与门店运营”

## 一个更实际的落地路径

如果要快速落地，不建议一开始就做全链路大系统，建议按下面顺序推进：

1. 先把真实菜品库建起来
2. 再把菜单推荐页接到真实菜品库
3. 然后把后厨调度页接到真实订单流
4. 最后逐步引入 AI 推荐、AI 文案和 AI 排程优化

这样做的好处是：

- 风险更低
- 更容易分阶段验收
- 更容易和门店现有系统对接
- 前期就能形成可展示、可试点、可迭代的版本
